Day 1 Week 1 · SQL Foundations

SELECT & WHERE

Today you learn the two most important SQL commands. By the end of today, you can answer real questions from your manager using data.

📖

Concept: What is SELECT?

10 min read

Think of a database like an Excel sheet. A table has rows (records) and columns (fields). SELECT is how you ASK the database to show you data.

SELECT tells the database: "Show me these columns." FROM tells it: "From which table?"

-- BASIC STRUCTURE
SELECT column1, column2
FROM table_name;

-- * means "show me ALL columns"
SELECT * FROM tickets;
💡

The semicolon ; at the end tells SQL "the query is finished." Think of it like a full stop at the end of a sentence.

📖

Concept: What is WHERE?

WHERE is a filter. It says "only show me rows where this condition is true." Without WHERE, you get ALL the data. With WHERE, you get exactly what you need.

-- WHERE EXAMPLES
-- Show only open tickets
SELECT * FROM tickets WHERE status = 'open';

-- Show only high priority tickets
SELECT ticket_id, title, priority
FROM tickets
WHERE priority = 'high';

-- Combine two conditions with AND
SELECT * FROM tickets
WHERE status = 'open' AND priority = 'high';

The = sign checks for exact match. Use quotes around text values. Numbers don't need quotes.

Day 2Week 1

ORDER BY & LIMIT

Today you learn to sort data and control how many rows you get back. These two commands are in almost every real-world query you'll ever write.

📖

Concept: ORDER BY — Sort Your Results

ORDER BY sorts the results. Think of it like clicking a column header in Excel to sort. ASC = A to Z (low to high). DESC = Z to A (high to low).

-- Sort tickets oldest first (ASC is default)
SELECT ticket_id, title, created_at
FROM tickets
ORDER BY created_at ASC;

-- Sort by priority, most urgent last (for alphabetical: high, low, medium)
SELECT * FROM tickets ORDER BY priority DESC;
📖

Concept: LIMIT — Control Result Size

LIMIT says "only show me the first N rows." Essential when your database has thousands of rows and you only need a summary or top results.

-- Show only the 3 most recent tickets
SELECT ticket_id, title, created_at
FROM tickets
ORDER BY created_at DESC
LIMIT 3;

-- Top 5 oldest OPEN tickets (most urgent to close!)
SELECT ticket_id, title, priority, created_at
FROM tickets
WHERE status = 'open'
ORDER BY created_at ASC
LIMIT 5;
💡

The ORDER of SQL clauses matters: SELECT → FROM → WHERE → ORDER BY → LIMIT. You can't put LIMIT before WHERE.

Day 3Week 1

DISTINCT & COUNT

Today: remove duplicates and start counting. These are the foundations of every report you'll ever build.

📖

DISTINCT — Remove Duplicate Values

DISTINCT returns only unique values. If 10 tickets all have status='open', DISTINCT only shows 'open' once.

-- What categories exist in our tickets?
SELECT DISTINCT category FROM tickets;

-- What priority levels do we have?
SELECT DISTINCT priority FROM tickets ORDER BY priority;
📖

COUNT — Count Rows

COUNT counts how many rows match your query. This is the most common reporting function in SQL.

-- How many total tickets?
SELECT COUNT(*) FROM tickets;

-- How many OPEN tickets?
SELECT COUNT(*) AS open_tickets
FROM tickets
WHERE status = 'open';

-- AS gives your result column a friendly name
💡

AS creates an alias — a friendly name for your result column. Instead of "COUNT(*)" you get "open_tickets" in your output. Always use aliases in real reports so managers understand the output.

Day 5Week 1

LIKE, IN & BETWEEN

Advanced filtering — search for patterns, multiple values, and ranges. These appear in almost every support database query.

📖

LIKE — Search by Pattern

LIKE searches for text patterns. Use % as a wildcard (means "any characters here").

-- Find all tickets about VPN (contains "VPN" anywhere)
SELECT * FROM tickets WHERE title LIKE '%VPN%';

-- Find tickets starting with "Password"
SELECT * FROM tickets WHERE title LIKE 'Password%';

-- Find tickets ending in "request"
SELECT * FROM tickets WHERE title LIKE '%request';
📖

IN & BETWEEN

-- IN: match any of these values (cleaner than multiple OR)
SELECT * FROM tickets
WHERE priority IN ('high', 'medium');

-- BETWEEN: find dates or numbers in a range
SELECT * FROM tickets
WHERE created_at BETWEEN '2024-01-08' AND '2024-01-10';

Day 6Week 1

GROUP BY & HAVING

The most powerful reporting tool in SQL. GROUP BY lets you summarize data by category — like "how many tickets per agent?" or "how many issues per category?"

📖

GROUP BY — Summarize by Category

GROUP BY groups rows that have the same value in a column, then lets you run aggregate functions (COUNT, SUM, AVG) on each group separately.

-- How many tickets per category?
SELECT category, COUNT(*) AS total_tickets
FROM tickets
GROUP BY category
ORDER BY total_tickets DESC;

-- How many tickets per agent?
SELECT agent_id, COUNT(*) AS ticket_count
FROM tickets
GROUP BY agent_id
ORDER BY ticket_count DESC;

-- HAVING filters AFTER grouping (WHERE filters before)
SELECT agent_id, COUNT(*) AS ticket_count
FROM tickets
GROUP BY agent_id
HAVING COUNT(*) > 2;
-- Only show agents with MORE than 2 tickets
💡

WHERE vs HAVING rule: WHERE filters rows BEFORE grouping. HAVING filters groups AFTER grouping. Use WHERE to filter raw data, HAVING to filter the grouped summary results.

SQL Interview Questions

Q1: What is the difference between WHERE and HAVING? ▼
WHERE filters individual rows BEFORE the data is grouped. HAVING filters group results AFTER GROUP BY runs. You can't use aggregate functions (COUNT, SUM) in WHERE — use HAVING for that.

Example: WHERE status = 'open' filters rows. HAVING COUNT(*) > 5 filters groups that have more than 5 rows.

Interviewer tip: Saying "WHERE runs before grouping, HAVING after" in the first sentence shows you really understand it.
Q2: What is the difference between INNER JOIN and LEFT JOIN? ▼
INNER JOIN returns only rows that have a match in BOTH tables. LEFT JOIN returns all rows from the left table, plus matching rows from the right table — if there's no match, the right table columns are NULL.

Real example: "LEFT JOIN agents ON tickets.agent_id = agents.agent_id" — gives you all tickets, even if no agent is assigned yet. INNER JOIN would hide unassigned tickets.

Use LEFT JOIN when you don't want to lose data from your main table.
Q3: Write a query to find duplicate records in a table. ▼
-- Find duplicate ticket titles
SELECT title, COUNT(*) AS occurrences
FROM tickets
GROUP BY title
HAVING COUNT(*) > 1
ORDER BY occurrences DESC;
This is one of the most common SQL interview tasks. GROUP BY + HAVING COUNT(*) > 1 is the standard pattern.
Q4: How would you find all tickets unresolved for more than 7 days? ▼
-- SQLite version
SELECT ticket_id, title, created_at
FROM tickets
WHERE status = 'open'
  AND julianday('now') - julianday(created_at) > 7;
This is a classic SLA breach detection query. Even if you don't know the date function by heart, saying "I'd use date functions to compare today's date with created_at" shows the right thinking.
Q5: What does SELECT DISTINCT do? Give an example. ▼
DISTINCT removes duplicate values from the result set. If you SELECT category FROM tickets and 10 rows have category='Network', you'll see 'Network' 10 times. DISTINCT shows it once.

Example: SELECT DISTINCT category FROM tickets; gives you a clean list of all unique ticket categories.
